# CLAUDE.md — Project instructions

Durable project rules for AI agents. **`AGENTS.md` is generated by Laravel Boost**
(`boost:update` runs on composer operations and overwrites it) — do NOT hand-edit
`AGENTS.md`; put lasting guidance here or in `docs/`.

## Code standards (enforced)

- **Frontend (JS / TS / TSX / JSON): Biome.** Run `bun run check`
  (`biome check --write .`) before finalizing any frontend change — it formats,
  organizes imports, and applies safe lint fixes. Verify with `bun run check:ci`
  (must be **0 errors**). Biome **replaced ESLint + Prettier** — do not
  reintroduce them. Full policy: [`docs/CODING-STANDARDS.md`](docs/CODING-STANDARDS.md).
- **UI / UX: shadcn + branded, not plain forms.** All UI must use the design
  tokens (indigo brand + neutral surfaces, light **and** dark), reach for
  shadcn/ui components, and handle every state (loading / empty / error / success)
  with real feedback. Never ship a bare black-and-white form. Full policy:
  [`docs/UI-UX-GUIDELINES.md`](docs/UI-UX-GUIDELINES.md)
  (agent digest: [`.ai/guidelines/ui-ux.md`](.ai/guidelines/ui-ux.md)).
- **Vendored shadcn/ui is READ-ONLY.** Never hand-edit
  `resources/js/components/ui/**` (or other generated trees:
  `resources/js/{routes,actions,wayfinder}`, `bootstrap/ssr`, `resources/css`).
  Editing a primitive to fix one screen reliably breaks others. Instead: pass
  props/className, **wrap** the primitive in your own `components/` file, or
  compose `radix-ui` directly — never fork `ui/`. Full rule:
  [`docs/CODING-STANDARDS.md`](docs/CODING-STANDARDS.md).
- **PHP: Laravel Pint.** Run `vendor/bin/pint --dirty` before finalizing PHP changes.
- **Types:** `bun run types:check` must pass.
- **Package manager / JS runtime: Bun** (`bun install`, `bun run …`; lockfile
  `bun.lock`). Not npm / pnpm / yarn.

## Testing

Three layers — run the ones your change touches:

- **PHP: Pest.** `php artisan test --compact` (or `--filter=Name`). Built TDD per the
  plans in `docs/superpowers/plans/` — failing test first, then implement.
- **TS / React: Vitest + Testing Library.** `bun run test:ci` (`bun run test` = watch).
  `lib/` logic + component/page render-smoke via the `resources/js/test/renderPage`
  helper (a global `@inertiajs/react` mock + jsdom polyfills live in `vitest.setup.ts`).
- **End-to-end: Playwright.** `bun run test:e2e` (real browser, boots the SSR build).
  A few critical flows only; catches SSR/hydration/routing bugs jsdom can't.

**Conventions these bugs taught us — follow them:**

- **Contract tests, not happy-path.** Test every param / filter / sort / pagination /
  empty path a screen can trigger — backend hits the real query params, frontend renders
  data **and** empty **and** filtered. Defaults passing ≠ the feature works.
- **SSR determinism.** No `Date.now()`, `Math.random()`, or unpinned
  `Intl`/`toLocaleString` in render output — compute in `useEffect` or pin the locale
  (see `NUMBER_LOCALE` in [`resources/js/lib/format.ts`](resources/js/lib/format.ts)).
  Nondeterministic render text = a React #418 hydration mismatch.
- **Dates use `Carbon\CarbonInterface`.** The app runs `Date::use(CarbonImmutable::class)`,
  so a concrete `Illuminate\Support\Carbon` hint is a latent runtime-type bug; type date
  params/returns as `CarbonInterface` and advance cursors by reassignment.
- **Server-side Inertia.** Expensive `Inertia::render` props should be closures
  (`fn () => …`) so partial `only:` reloads don't recompute them; `defer()` heavy
  analytics props; keep `HandleInertiaRequests::share()` minimal + typed.

## Architecture

See `docs/superpowers/specs/` (master spec) and `docs/superpowers/plans/`.

- Multi-tenant manufacturing/inventory ERP: Laravel 13 + Inertia v3 + React 19 +
  TypeScript, Tailwind v4. `stancl/tenancy` in **multi-database** mode,
  tenants identified by **URL slug** (`/{slug}/…`).
- One central (landlord) DB (super-admins + `organizations`) + one DB per tenant
  (that tenant's `users` + business tables; no `organization_id` columns).
- The central DB connection is named **`central`**. **Never** define a connection
  literally named `tenant`.
- Central migrations live in `database/migrations/`; per-tenant migrations in
  `database/migrations/tenant/`. Keep them straight.

## Pre-finalize checklist

A **`lefthook`** gate enforces this automatically (fast checks on commit, full suite on
push; installed by `bun run prepare`). Run it by hand too before finalizing:

1. `bun run check` (frontend changed) · `vendor/bin/pint --dirty` (PHP changed)
2. `bun run types:check` (TS) · `composer types:check` (PHPStan level 7, PHP changed)
3. `bun run test:ci` (frontend changed) · `php artisan test --compact` (affected)
4. `bun run build` before a release/deploy

## Review flow

Two layers — deterministic first, judgment second:

- **Deterministic (free, automatic).** The lefthook gates above ARE the review for
  mechanical issues: Biome + Pint (format/lint, a11y/correctness at error for authored
  code), `tsc` + **PHPStan** (types), **Pest arch tests** (`tests/Arch` — strict types,
  no debug helpers, DTO/Request/enum shape, no `tenant` connection), Vitest + Pest
  (behaviour), the `ui-guard` (design tokens, read-only trees, SSR determinism), and the
  production build. Prefer encoding a new rule here over relying on a human/agent.
- **Judgment (cheap, on demand).** For the semantic/UX bugs the gates can't catch (wrong
  copy, a misreported state, a broken edge case), run the cheap diff-scoped review:
  `Workflow({ scriptPath: '.claude/workflows/diff-review.js' })` — one reviewer per changed
  file, single pass, skips small diffs. For a deeper, billed pass on a risky change, use
  **`/code-review`** (local branch) or `/code-review ultra` (cloud). Don't spin up bespoke
  multi-agent review swarms — they're expensive; reach for these instead.
